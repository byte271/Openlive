# Openlive v1.2.0 — `web/` Junk Code Audit

**Task ID:** `2-audit`
**Scope:** `apps/openlive-gateway/web/` (12 files)
**Auditor:** general-purpose agent
**Date:** 2026
**Status:** Audit only — no files modified

## Methodology

Every file in scope was read in full. Cross-file references were verified with
ripgrep to distinguish genuinely dead code from merely non-obvious usage. Each
finding below lists the file, line number(s), category, concrete description,
and a recommended action. Findings are grouped by file and ordered by severity
within each file.

Severity legend:

- **P1** — unambiguous dead code or redundant logic that should be removed/merged.
- **P2** — likely junk but context-dependent; confirm before acting.
- **P3** — cosmetic / consistency nit; optional.

## `app.js`

### A1. `parent_event_id` always serialized as `null` — P2
- **Lines:** 360
- **Category:** redundant logic / dead field
- **Description:** In `sendControl`, the envelope always sets
  `parent_event_id: null`. No code path ever sets it to a real value. It is
  either a placeholder for future protocol support or a vestigial field. If the
  gateway protocol never uses it, the line is dead; if it is reserved for
  future use, it should be promoted to a `sendControl` parameter with a
  `null` default so callers can actually populate it.
- **Action:** Confirm against the gateway protocol. Either remove the line, or
  add a `parentEventId = null` parameter to `sendControl` and thread it through.

### A2. `mediaTimeUs` increment hardcoded to `20_000` — P2
- **Lines:** 281
- **Category:** redundant logic / tight coupling
- **Description:** `sendAudioFrame` does `mediaTimeUs += 20_000`, hardcoded to
  20 ms. The same function passes `frameDurationMs: 20` to `encodeInputAudio`
  two lines above. If `frameDurationMs` ever changes, `mediaTimeUs` will drift
  silently. The two should share a single `FRAME_DURATION_MS` constant.
- **Action:** Introduce a module-level `const FRAME_DURATION_MS = 20;` (or
  import from `protocol.js`) and use it in both places.

### A3. Duplicated `inputSampleRate` state — P2
- **Lines:** 38 (module var) vs. `audio-session.js:12` (instance var)
- **Category:** duplicated state
- **Description:** `app.js` keeps `let inputSampleRate = 16000;` and
  `audio-session.js` keeps `this.inputSampleRate = 16000;`. Both are updated
  together in `handleControl` (`inputSampleRate = payload.input_sample_rate;
  audio.setInputSampleRate(inputSampleRate);`). The app.js copy is only read in
  `sendAudioFrame` to populate the encoded packet; it could instead query
  `audio.inputSampleRate` (which would require exposing it) or the duplication
  could be documented as intentional. Today it is a silent sync hazard.
- **Action:** Either expose a getter on `AudioSession` and drop the app.js
  copy, or add a comment explaining why both are kept.

### A4. `resetExperience()` bypasses the `transition()` guard — P2
- **Lines:** 87, 161 → `ui.js:160-167`
- **Category:** redundant state tracking / subtle desync
- **Description:** Voice mode is tracked in three places: `app.js` `mode`,
  `ui.js` `document.body.dataset.mode`, and `VoiceVisualizer.mode`. The
  `transition()` function keeps all three in sync. But `resetExperience()`
  (called from app.js init and `endConversation`) calls `setVoiceMode(IDLE)`
  directly, updating only the DOM — it does **not** reset `app.js`'s `mode`
  variable. After ending a conversation, `mode` retains its last value (e.g.
  `SPEAKING`). If the user then starts a new conversation and `transition` is
  called with the same value, the guard `if (nextMode === mode && detail ===
  undefined) return;` could short-circuit incorrectly. The visualizer is
  explicitly reset (`visualizer.setMode(VoiceMode.IDLE)`) right after, but
  `app.js`'s `mode` is not.
- **Action:** Either call `transition(VoiceMode.IDLE)` inside
  `resetExperience` (and remove the separate `visualizer.setMode(IDLE)` calls),
  or set `mode = VoiceMode.IDLE` in `endConversation` / module init.

### A5. `pointerdown` outside-click handler only closes settings — P3
- **Lines:** 78-85
- **Category:** inconsistent logic
- **Description:** The global `pointerdown` listener closes the settings
  popover on outside click but not the diagnostics drawer. The `Escape` handler
  (`closeOverlays()`) closes both. The asymmetry is plausibly intentional
  (diagnostics is a large drawer the user may want to keep open while clicking
  elsewhere), but it is not documented.
- **Action:** Leave as-is, but add a one-line comment explaining the intent, or
  make the behavior consistent.

### A6. Brand button `aria-label` says "Open diagnostics" — P3
- **Lines:** index.html:14 + app.js:73
- **Category:** other (UX mismatch)
- **Description:** The brand button displays the Openlive logo/name but its
  `aria-label` and click handler both open diagnostics. This is a hidden
  affordance, not junk code, but worth flagging because it surprises screen
  readers.
- **Action:** Consider giving the brand button a neutral label and routing
  diagnostics only through the dedicated debug button.

## `ui.js`

### U1. `toggleSettings` / `toggleDiagnostics` return values never used — P2
- **Lines:** 128-132, 134-140
- **Category:** dead return logic
- **Description:** Both functions compute and `return next;` (a boolean).
  Every call site in `app.js` and `ui.js` (`closeOverlays`, the `pointerdown`
  handler, the click handlers) discards the return value. The return is dead
  code.
- **Action:** Either drop the `return next;` statements, or — if a future
  caller needs the toggled state — keep them and add a comment.

### U2. `clamp` helper duplicated — P1
- **Lines:** 169-171
- **Category:** duplicated helper
- **Description:** `clamp(value)` is defined identically in `ui.js` (here) and
  `visual-state.js` (lines 81-83), both with the `Number.isFinite` guard. A
  third, simpler `clamp` (no guard) lives in `voice-visualizer.js`
  (lines 139-141). Three copies of the same primitive.
- **Action:** Move `clamp` into `audio-utils.js` (or a new `util.js`) and
  import from there. Unify the implementation (keep the `Number.isFinite`
  guard).

### V1. `signalEnergy` only differentiates two modes — P3
- **Lines:** 71-75
- **Category:** other (simplification opportunity)
- **Description:** `inputWeight` is `0.92` for `LISTENING` and `0.58` for

  readers do not expect per-mode weighting.
- **Action:** Leave, or refactor to explicit `{ input, output }` weights keyed
  off two booleans for clarity.

- **Lines:** 81-83
- **Category:** duplicated helper
- **Description:** Identical to `ui.js`'s `clamp`. See U2.

### VV1. `destroy()` method never called — P1
- **Lines:** 38-40
- **Category:** dead code
- **Description:** `VoiceVisualizer.destroy()` calls
  `cancelAnimationFrame(this.frame)` to stop the render loop. No code in
  `app.js` (or anywhere else) ever calls `destroy()`. The visualizer runs for

## `audio-session.js`

### AS1. `isMicrophoneActive()` method never called — P1
- **Lines:** 26-28
- **Category:** dead code
- **Description:** The method `isMicrophoneActive()` returns
  `Boolean(this.microphoneStream)`. No caller exists — `app.js` tracks
  microphone state with its own `microphoneActive` boolean instead.
- **Action:** Remove the method, or refactor `app.js` to use it and drop the
  duplicated `microphoneActive` flag.

### AS2. `reset()` and `cancel()` duplicate the cancel-message logic — P2
- **Lines:** 68-83 (`reset`) vs. 124-140 (`cancel`)
- **Category:** duplicated logic
- **Description:** `reset()` iterates `playbackFrameCounts.keys()` and posts
  `{type: "cancel", generationId, fadeMs: 25}` to the playback worklet for
  each. `cancel()` posts the same shape (with a configurable `fadeMs`) for a
  single generation. The two paths do not share a helper. `reset()` also
  bypasses `canceledGenerations` bookkeeping, which is intentional (full
  reset) but undocumented.
- **Action:** Extract a private `postCancel(generationId, fadeMs)` helper and
  call it from both. Document why `reset()` skips `canceledGenerations`.

### AS3. `canceledGenerations` 256-entry eviction loop is defensive dead weight — P3
- **Lines:** 127-131
- **Category:** unreachable branch / defensive code
- **Description:** The `while (this.canceledGenerations.size > 256)` loop
  evicts the oldest entry once the set exceeds 256. In practice the set holds
  one entry per cancelled generation, and the playback worklet's
  `completeGenerations` is the parallel structure. Reaching 256 cancelled
  generations without a reset would require an extreme session. The cap is
  arbitrary and the branch is effectively unreachable.
- **Action:** Leave (cheap insurance) or remove and rely on `reset()` to clear
  the set. If kept, add a comment explaining the 256 choice.

### AS4. `applyLocalInterruption` recomputes a value app.js also computes — P3
- **Lines:** 195-201 (`targetSpeechProbability`) vs. `app.js:285`
- **Category:** redundant logic
- **Description:** `handleCaptureFrame` computes
  `targetSpeechProbability = speechProbability * (1 - echoProbability)` and
  passes it to `applyLocalInterruption`. `app.js:updateInputActivity`
  independently computes `inputLevel = speechProbability * (1 - echoProbability)`.
  Same formula, two locations.
- **Action:** Optional: have `onInputActivity` receive the already-combined
  probability, or factor the formula into a shared helper.

## `protocol.js`

### P1. `decodeOutputAudio` returns fields no consumer reads — P1
- **Lines:** 48-49 (`frameDurationMs`, `channels`)
- **Category:** dead decoded data
- **Description:** `decodeOutputAudio` returns an object with `frameDurationMs`
  and `channels`. Callers are `app.js:handleMedia` (which only forwards
  `sequence` to `observeServerSequence` and passes the rest to
  `audio.enqueue`) and `audio-session.js:enqueue` (which reads `generationId`,
  `pcm`, `sampleRate`, `mediaTimeUs` only). The test
  (`tests/protocol.test.js`) asserts on `sequence`, `mediaTimeUs`,
  `generationId`, `pcm` — never on `frameDurationMs` or `channels`. Both
  fields are decoded and immediately discarded.
- **Action:** Remove `frameDurationMs` and `channels` from the returned
  object. (If the protocol gains a consumer later, restore them then.)

### P2. `encodeInputAudio` hardcodes `channels = 1` via `setUint8(46, 1)` — P3
- **Lines:** 26
- **Category:** other (implicit parameter)
- **Description:** The encoder writes `1` to byte 46 (channels) unconditionally
  rather than threading a `channels` parameter. This is fine — the capture
  path is mono by construction (`channelCount: 1` in `getUserMedia`) — but the
  asymmetry with `decodeOutputAudio` (which *does* read channels) is a minor
  smell.
- **Action:** Leave, but if P1 is acted on (drop `channels` from decode), also
  drop the read on the decode side and document byte 46 as "reserved, always
  1".

## `styles.css`

### S1. `--accent-warm` CSS variable set but never read — P1
- **Lines:** 11 (`:root`), 462, 467, 472, 477, 483 (`body[data-mode="…"]`)
- **Category:** dead CSS variable
- **Description:** `--accent-warm` is declared in `:root` and overridden in
  five `body[data-mode]` selectors (listening, thinking, speaking,
  interrupted, error/connection_error). A grep for `var(--accent-warm)`
  returns **zero** reads. Only `var(--accent)` is consumed (once, in
  `.orb-glow` line 179). The warm-accent variable is pure dead weight: six
  declarations producing no visual effect.
- **Action:** Remove all `--accent-warm` declarations, or wire it into
  `.orb-glow` (e.g. as a second stop in the `color-mix`) so the per-mode
  overrides actually do something.

### S2. `body[data-mode]` rules missing for 4 of 10 modes — P3
- **Lines:** 460-484
- **Category:** other (incomplete theming)
- **Description:** `VoiceMode` has 10 states; CSS overrides `--accent` /
  `--accent-warm` for only 6 (`listening`, `thinking`, `speaking`,
  `interrupted`, `error`, `connection_error`). The remaining 4 (`idle`,
  `starting`, `muted`, `reconnecting`) fall through to the `:root` defaults.
  Not dead code, but the gap is worth noting — especially `reconnecting` and
  `muted`, which arguably deserve distinct accent colours.
- **Action:** Optional: add `body[data-mode="reconnecting"]` and
  `body[data-mode="muted"]` rules. (If S1 is acted on by removing
  `--accent-warm`, these new rules only need `--accent`.)

### S3. `.sheet-handle` shown only on mobile — leave
- **Lines:** 376-378, 526-533
- **Category:** (no action)
- **Description:** `.sheet-handle` is `display: none` on desktop and
  `display: block` inside the `max-width: 600px` media query. The element
